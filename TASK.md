## Общая оценка

**Сейчас llama-launcher — уже не сырой прототип, а крепкий pre-release уровня примерно 7/10.**
Архитектура разумная: Tauri + Rust для системной части, SvelteKit для интерфейса, отдельные модули runtime, сервера, конфигурации, Hugging Face и автоподбора параметров. Продукт уже решает понятную задачу: скачать движок, подобрать backend, найти модель и запустить `llama-server`.

Особенно хорошо, что за последние коммиты ты уже закрыл много типичных критических проблем: атомарное сохранение настроек, race condition при автоконфигурации, валидацию запуска, RAM-aware настройки, проверку `/health`, безопасное скачивание и staging-установку.

Но до полноценного публичного **1.0** ещё не хватает нескольких фундаментальных слоёв.

# P0 — исправить до распространения среди пользователей

## 1. Автотест установки CUDA runtime на реальном архиве

Тот критический CUDA-баг, который ты описывал ранее, действительно находится в самой рискованной части приложения.

Сейчас CUDA-ассеты задаются жёстко:

```text
llama-b9963-bin-win-cuda-12.4-x64.zip
cudart-llama-bin-win-cuda-12.4-x64.zip
```

После распаковки код рекурсивно ищет `llama-server.exe`, но затем ожидает, что после копирования он окажется непосредственно в корне staging.

Это уже улучшено по сравнению с наивным поиском, однако нужен **интеграционный тест на настоящих ZIP-файлах llama.cpp**:

* CPU archive → найден сервер;
* Vulkan archive → найден сервер;
* CUDA archive + cudart → сервер и DLL лежат вместе;
* smoke test запускается;
* после установки `runtime_status()` возвращает правильную директорию;
* установленный бинарник реально стартует без отсутствующих DLL.

Unit-тестов отдельных функций недостаточно. Самая важная вещь — проверить полный pipeline:

```text
release metadata
→ download
→ checksum
→ extract
→ merge cudart
→ locate executable
→ smoke test
→ activate
→ launch server
```

**Это главный следующий технический шаг.**

## 3. Включить CSP

В Tauri сейчас:

```json
"security": {
  "csp": null
}
```

Для десктопного приложения, которое:

* обращается к Hugging Face;
* открывает внешние ссылки;
* управляет локальным процессом;
* принимает данные из API;
* потенциально отображает пользовательские строки,

отсутствие Content Security Policy — нежелательная поверхность атаки.

Нужно определить явный CSP примерно по принципу:

```text
default-src 'self'
connect-src 'self' https://huggingface.co https://*.huggingface.co
img-src 'self' data: https:
style-src 'self' 'unsafe-inline'
font-src 'self' data:
```

Фактический список надо подобрать после проверки Svelte/Tauri и всех сетевых запросов.

## 4. Не заявлять поддержку платформ, которых фактически нет

Tauri настроен на:

```json
"targets": "all"
```

Но managed runtime явно ориентирован на Windows x64:

* бинарник `llama-server.exe`;
* CUDA/Vulkan Windows ZIP;
* `LOCALAPPDATA`;
* DXGI;
* `taskkill`;
* Windows API для места на диске.

Поэтому сейчас лучше честно сделать:

```json
"targets": ["nsis", "msi"]
```

или только тот формат, который реально тестируется.

macOS/Linux добавлять позже отдельной задачей. Иначе сборка может формально создаться, но основная функциональность будет неработоспособна.

# P1 — нужно для качественного публичного релиза

## 5. Нормальная система обновления llama.cpp

Pinned runtime — хорошее и безопасное решение: фиксированный тег и SHA-256 защищают от внезапного изменения архивов.

Но сейчас обновление движка требует нового релиза всего приложения.

Нужно разделить:

```text
Версия LlamaLauncher
Версия managed llama.cpp runtime
```

Рекомендованная модель:

* приложение содержит подписанный runtime manifest;
* manifest содержит тег, имена ассетов, SHA-256 и совместимость;
* приложение может получить новый manifest с твоего GitHub Release;
* пользователь видит «Доступно обновление движка»;
* старый runtime остаётся до успешного smoke test;
* возможность отката на предыдущую версию.

Важно: manifest должен быть подписан или поставляться вместе с обновлением приложения, иначе SHA из удалённого JSON не даёт полноценной защиты.

## 7. Подпись Windows-приложения

Без code signing Windows SmartScreen будет пугать пользователей неизвестным издателем.

Для серьёзного распространения понадобятся:

* Authenticode certificate;
* подпись `.exe`, NSIS/MSI;
* checksum на странице релиза;
* желательно SBOM;
* reproducible или хотя бы документированная сборка.

Это не обязательно для тестовых версий, но обязательно для ощущения настоящего продукта.

# P2 — продуктовые возможности

## 11. Встроенный чат вернуть, но правильно

По истории видно, что встроенный чат был добавлен и сразу откатан.

Само направление правильное: пользователь не должен запускать сервер, а потом гадать, куда идти.

Но чат лучше строить отдельным стабильным модулем:

* streaming tokens;
* stop generation;
* история диалогов;
* system prompt;
* temperature/top-p/max tokens;
* markdown;
* копирование кода;
* отображение tokens/s;
* context usage;
* выбор модели;
* сохранение чатов локально;
* OpenAI-compatible API как отдельный режим.

Но **не надо превращать launcher в аналог LM Studio сразу**. Сначала идеальный цикл:

```text
Скачал модель → запустил → задал первый вопрос
```

Расширенные параметры лучше скрывать за режимом Expert, который у тебя уже концептуально существует. История коммитов показывает, что уровни пользователя уже влияют на плотность интерфейса.

## Самое важное решение

**Сейчас не надо добавлять двадцать новых функций.**

Приложению уже хватает возможностей для хорошего первого продукта. Основная задача:

> Сделать так, чтобы на любом типичном Windows-компьютере пользователь гарантированно прошёл путь от скачивания приложения до первого ответа модели без терминала, ручного поиска файлов и непонятных ошибок.

Именно надёжность установки CUDA/Vulkan, диагностика и CI сейчас дадут проекту больше ценности, чем мультимодальность, плагины или ещё один редизайн.
